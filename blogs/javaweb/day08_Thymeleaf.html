# 第二章.Thymeleaf入门

# 1.Thymeleaf的简介

# 1.1 Thymeleaf的概念

1.概述:
  Thymeleaf是一款用于渲染XML/XHTML/HTML5内容的模板引擎。类似JSP，Velocity，FreeMaker等， 它也可以轻易的与Spring MVC等Web框架进行集成作为Web应用的模板引擎。
      
2.作用:
  在静态页面上动态显示服务器中返回的数据

# 1.2 Thymeleaf的优势

1.SpringBoot官方推荐使用的视图模板技术，和SpringBoot完美整合。SpringBoot后面会讲
2.不经过服务器运算仍然可以直接查看原始值，对前端工程师更友好。
  一个页面,如果没有通过tomcat运行,直接打开该页面,我们看到的就是一个正常的页面(静态)
  一个页面,如果通过tomcat运行,网页中的一些数据就会动态发生改变,展示在用户面前

<!DOCTYPE html>
<html lang="en" xmlns:th="http://www.thymeleaf.org">
<head>
    <meta charset="UTF-8">
    <title>Title</title>
</head>
<body>

    <p th:text="${username}">Original Value</p>

</body>
</html>

Thymeleaf最大优势:

如果一个页面上用了thymeleaf,但是不通过tomcat运行,thymeleaf不生效,但是页面能正常显示,显示原始内容

如果一个页面上用了thymeleaf,通过tomcat运行,thymeleaf自动生效,将页面上的原始静态数据变成了服务器返给页面的动态数据

# 1.3 物理视图

1.在Servlet中，将请求转发到一个HTML页面文件时，使用的完整的转发路径就是"物理视图"
  比如:/pages/user/login_success.html   
    
2.如果我们把所有的HTML页面都放在某个统一的目录下，那么转发地址就会呈现出明显的规律：
  比如:
   /pages/user/login.html
   /pages/user/login_success.html
   /pages/user/regist.html
   /pages/user/regist_success.html
      
3.路径的开头都是：/pages/user/      
  路径的结尾都是：.html
  所以，路径开头的部分我们称之为"视图前缀"，路径结尾的部分我们称之为"视图后缀"

# 1.4逻辑视图

物理视图=视图前缀+逻辑视图+视图后缀

上面的例子中：
视图前缀逻辑视图视图后缀物理视图
/pages/user/login.html/pages/user/login.html
/pages/user/login_success.html/pages/user/llogin_success.html

# 2.2 配置上下文参数

1.将来请求转发的页面都有可能在同一个路径下,所有的页面都是.html结尾,所有我们可以在web.xml中配置全局参数
  配置前缀和后缀
    
2.怎么配置前缀和后缀:
  前缀:就是我们页面所在在共同的路径
      如果页面都在WEB-INF/view下,那么前缀就配置WEB-INF/view/   -> 以后使用thymeleaf做请求转发,都会自动跑到该路径下
  后缀:
      .html
          
3.配置前缀和后缀的作用:
  以后请求转发就不用写路径和页面文件扩展名,直接写页面文件名即可
 
4.注意:前缀,后缀写的都是页面的共同的路径部分,可以根据实际情况进行修改

   <!--
      先配置前缀
        如果页面放到了WEB-INF下了,那么value就写WEB-INF对应的路径前缀
        如果param-value配置啥路径,那么使用thymeleaf做请求转发都会自动跑到该路径下
    -->
    <context-param>
        <param-name>view-prefix</param-name>
        <param-value>/WEB-INF/view/</param-value>
    </context-param>

    <!--
      配置后缀
      所有请求转发到的页面,后缀都是.html
    -->
    <context-param>
        <param-name>view-suffix</param-name>
        <param-value>.html</param-value>
    </context-param>

为什么要将页面放到WEB-INF下?

原因:WEB-INF下的资源浏览器访问不到,所以我们将页面以及静态资源放到目录下,是一种保护

比较不好的地方:

​ 1.浏览器直接访问不到页面,只能所有的请求先给servlet,让servlet将请求转发到WEB-INF下的资源上

​ 2.放到WEB-INF下,重定向咋整

​ 可以先重定向到servlet,在通过servlet转发到WEB-INF下

# 2.3 创建Servlet基类

这个类大家直接复制粘贴即可，将来使用框架后，这些代码都将被取代。

此类的作用:所有想使用Thymeleaf的Servlet都要继承此类才能使用Thymeleaf,然后将转发到指定的WEB-INF/view下

package com.atguigu.controller;

import org.thymeleaf.TemplateEngine;
import org.thymeleaf.context.WebContext;
import org.thymeleaf.templatemode.TemplateMode;
import org.thymeleaf.templateresolver.ServletContextTemplateResolver;

import javax.servlet.ServletContext;
import javax.servlet.ServletException;
import javax.servlet.annotation.WebServlet;
import javax.servlet.http.HttpServlet;
import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import java.io.IOException;

/*
  专门配置Thymeleaf模板
  让Thymeleaf知道将来请求转发到哪里->在web.xml中配置的路径
 */
public class ViewBaseServlet extends HttpServlet {
    //创建一个Thymeleaf模板对象
    private TemplateEngine templateEngine;

    @Override
    public void init() throws ServletException {

        // 1.获取ServletContext对象
        ServletContext servletContext = this.getServletContext();

        // 2.创建Thymeleaf解析器对象
        ServletContextTemplateResolver templateResolver = new ServletContextTemplateResolver(servletContext);

        // 3.给解析器对象设置参数
        // ①HTML是默认模式，明确设置是为了代码更容易理解
        templateResolver.setTemplateMode(TemplateMode.HTML);

        // ②设置前缀
        String viewPrefix = servletContext.getInitParameter("view-prefix");

        templateResolver.setPrefix(viewPrefix);

        // ③设置后缀
        String viewSuffix = servletContext.getInitParameter("view-suffix");


        templateResolver.setSuffix(viewSuffix);

        // ④设置缓存过期时间（毫秒）
        templateResolver.setCacheTTLMs(60000L);

        // ⑤设置是否缓存
        templateResolver.setCacheable(true);

        // ⑥设置服务器端编码方式
        templateResolver.setCharacterEncoding("utf-8");

        // 4.创建模板引擎对象
        templateEngine = new TemplateEngine();

        // 5.将页面格式,前缀,后缀等参数设置到解析器之后,然后将解析器设置到Thymeleaf模板对象中
        templateEngine.setTemplateResolver(templateResolver);

    }

    /*
      解析Thymeleaf模板
      只要是和有Thymeleaf的页面打交道的servlet都要调用此方法
      想转发到哪个页面,templateName就接收哪个页面的名字
      
      将来调用processTemplate("页面名",request,response)->直接可以实现将请求转发到  前缀/页面名.html 页面上
    */
    protected void processTemplate(String templateName, HttpServletRequest req, HttpServletResponse resp) throws IOException {
        // 1.设置响应体内容类型和字符集
        resp.setContentType("text/html;charset=UTF-8");

        // 2.创建WebContext对象
        WebContext webContext = new WebContext(req, resp, getServletContext());

        // 3.处理模板数据
        templateEngine.process(templateName, webContext, resp.getWriter());
    }
    
}

# 2.4 入门案例_利用Thymeleaf直接转发指定页面

1.在WEB-INF/view下创建target1.html
2.在创建一个servlet继承ViewBaseServlet
3.调用ViewBaseServlet中的processTemplate方法实现请求转发,到指定的target1.html页面

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <title>Title</title>
</head>
<body>
我是第一个Thymeleaf页面
</body>
</html>

@WebServlet("/thymeleaf1")
public class Thymeleaf01Servlet extends ViewBaseServlet {
    protected void doGet(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
       //请求转发到target1.html中
        processTemplate("target1",request,response);
    }

    protected void doPost(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
        doGet(request, response);
    }
}


小结:
  1.如果我们将所有的页面放到了WEB-INF/view下,那么将来我们请求转发地址都要写  WEB-INF/view/页面名.html
  2.我们发现规律:请求转发的路径中前面有一部分一样,后面所有页面的后缀名也一样
  3.根据规律,在web.xml中配置了前缀和后缀
    前缀:就是请求转发路径中所有页面相同的路径
    后缀:页面的扩展名(.html)
  4.引入了thymeleaf工具类,解析在web.xml中配置的前缀和后缀
  5.以后请求转发直接调用thymeleaf工具类中的 processTemplate 方法
    参数:请求转发的页面名,request,response

# 2.5 入门案例_利用Thymeleaf取出服务器数据

1.Thymeleaf最常用的基本语法格式:
  th:text = "${域对象的key}" 可以将域对象中的值放到标签体中
2.注意:
  a.如果标签体中有数据,使用thymeleaf之后,原来的标签体会被thymeleaf从域中取出来的值替换
  b.如果标签中有数据,然后${域对象的key}中的key名写错了,标签体中原来的数据会被""代替
      
3.需求:
  利用Thymeleaf请求转发到target2.html中,在target2.html中去出服务器的数据

<!DOCTYPE html>
<html lang="en" xmlns:th="http://www.thymeleaf.org">
<head>
    <meta charset="UTF-8">
    <title>Title</title>
</head>
<body>
<div th:text = "${username}">我是div中原有的标签体内容</div>
</body>
</html>

@WebServlet("/thymeleaf2")
public class Thymeleaf02Servlet extends ViewBaseServlet {
    protected void doGet(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
        //往请求域中存数据,在页面上获取
        request.setAttribute("username","root");
        //将域中的数据请求转发到指定页面上
        processTemplate("target2",request,response);
    }

    protected void doPost(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
        doGet(request, response);
    }
}

# 1.3 解析URL地址

1.格式:
  th:href = "@{/项目下的某个资源}"
2.作用:

3.注意:
  其中的@符号会被Thymeleaf解析成当前web应用程序名字     

<!DOCTYPE html>

</head>
<body>
<!--
  @会被thymeleaf解析成当前web应用程序名称
  @{/index.html}->相当于-> 项目名/index.html
-->
<a th:href="@{/index.html}">访问index.html</a>
</body>
</html>

# 4 内置对象(了解)

# 4.1 内置对象的概念

1.概述:所谓内置对象其实就是在Thymeleaf的表达式中可以"直接使用"的对象

# 4.2 基本内置对象

- 调用方法时需要传参的也可以直接传入参数

1.问题描述:
  如果我们对"商品"进行增删改查,调用"商品"对应的方法,创建商品对应的servlet
  如果我们对"订单"进行增删改查,调用"订单"对应的方法,创建订单对应的servlet
  此时我们发现,多个servlet中都需要写反射的那一坨代码,动态的获取要执行的方法,反射的代码重复了
      
2.问题解决:
  可以创建一个servlet的父类,叫BaseServlet
  将所有servlet中反射的那一坨代码抽取出来
      
  让其他的servlet继承BaseServlet即可

public class BaseServlet extends ViewBaseServlet {
    protected void doGet(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
       //处理请求和响应中文乱码
        request.setCharacterEncoding("utf-8");
        response.setContentType("text/html;charset=utf-8");

        //1.获取请求参数->其实就是获取的一个方法名
        String methodName = request.getParameter("method");
        //2.获取当前类的class对象
        Class classServlet = this.getClass();
        //3.根据获取出来的methodName获取对应的方法
        try {
            Method method = classServlet.getMethod(methodName, HttpServletRequest.class, HttpServletResponse.class);
            //4.执行方法
            method.invoke(this,request,response);
        } catch (Exception e) {
            e.printStackTrace();
        }
    }

    protected void doPost(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
        doGet(request, response);
    }
}
